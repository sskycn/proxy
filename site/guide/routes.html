路由规则

用 route.json 控制哪些目标必须走上游。

tcptun 对 TCP 目标默认直连优先，失败后学习并写回。你也可以用 force_upstream 规则提前指定域名、后缀、正则、IP 和 CIDR。

优先级

force_upstream 规则优先于直连探测。命中规则时，目标直接走上游或 client/server 隧道。

完整 route.json 示例

{
  "force_upstream": {
    "domains": ["example.com", "api.example.net"],
    "domain_regexes": ["(^|\\\\.)example-cdn\\\\.com$"],
    "domain_suffixes": ["google.com", "github.com"],
    "ip_cidrs": ["8.8.8.0/24", "2001:4860::/32"],
    "ip_ranges": ["1.1.1.0/24"],
    "ips": ["9.9.9.9"]
  }
}

字段都可以为空数组。建议保留所有字段，后续人工维护和程序写回都更直观。

每种规则怎么写

字段匹配方式例子使用建议
domains精确域名api.example.com只想让一个明确 host 走上游时使用。
domain_suffixes后缀及其子域名github.com想让整个站点族走上游时使用，通常比正则更快更清晰。
domain_regexesGo/RE2 正则(^|\\.)example-cdn\\.com$复杂模式才使用，注意 JSON 里反斜杠要转义。
ips精确 IP9.9.9.9目标是固定 IP 时使用。
ip_cidrsCIDR 前缀8.8.8.0/24IP 段走上游，支持 IPv4 和 IPv6。
ip_rangesCIDR 风格别名1.1.1.0/24解析方式与 ip_cidrs 相同。

命令行预置 CIDR

tcptun config 可以在生成 route.json 时预置 IP CIDR。

tcptun config \
  --server-addr proxy.example.com:9443 \
  --force-ip-cidrs 8.8.8.0/24,2001:4860::/32

直连优先与自动学习

如果目标没有命中强制上游规则，TCP 流量会先尝试直连：

  • 普通 HTTP 请求需要在 direct_probe_timeout 内收到直连目标返回的首字节。
  • HTTP CONNECT 和 SOCKS5 CONNECT 会等待客户端首个隧道数据包，发给直连目标后再等待响应。
  • 探测失败时，该目标会标记为仅走上游，并在退出时合并写入 route.json。
  • UDP 规则更保守：内网 UDP 目标直连，其他 UDP 目标走上游。
{
  "mode": "client",
  "direct_probe_timeout": "800ms"
}

写回和去重

程序退出前会把学习到的直连失败目标合并进 route.json。已有规则会去重。某个可注册主域名下超过 3 个子域名进入 domains 后，会提升为 domain_suffixes，减少规则数量。

如果不希望写回，传入空路由路径禁用路由加载和写回：

tcptun client --config client.json --route-config ""

场景示例

让常用开发服务走上游

{
  "force_upstream": {
    "domains": ["registry.npmjs.org"],
    "domain_regexes": [],
    "domain_suffixes": ["github.com", "githubusercontent.com"],
    "ip_cidrs": [],
    "ip_ranges": [],
    "ips": []
  }
}

让指定 DNS 服务 IP 段走上游

{
  "force_upstream": {
    "domains": [],
    "domain_regexes": [],
    "domain_suffixes": [],
    "ip_cidrs": ["1.1.1.0/24", "8.8.8.0/24"],
    "ip_ranges": [],
    "ips": ["9.9.9.9"]
  }
}
